# Как называется такой подход

То, что ты описываешь, — это не один визуальный стиль типа “glassmorphism”, а метод проектирования:

**1. Information Architecture / IA**  
Информационная архитектура: как устроено дерево сайта, разделы, вложенность, навигация.

**2. Information-dense UX / Dense Minimalism**  
Информационно-плотный интерфейс: максимум смысла на минимальной площади, но без хаоса.

**3. Functional Minimalism**  
Функциональный минимализм: минимум декора, максимум полезности.

**4. Content-first Design**  
Сначала контент и задачи пользователя, потом красота.

**5. Progressive Disclosure**  
Постепенное раскрытие: сначала главное, детали — по клику, раскрытию, вкладке, фильтру.

Я бы назвал твою систему так:

> **IA-first Dense Minimalism**  
> или по-русски: **информационно-плотный функциональный минимализм с архитектурой от задач пользователя**.

А набор правил называется:

> **Design System + UX Guidelines + Design Principles + Design Tokens + Information Architecture Rules**

# Дизайн-система v2.0  
## «Интуитивная плотность»

Главная цель:

> **Минимум места. Максимум понятной информации. Минимум мыслительного усилия.**

Но важно:  
**плотно ≠ тесно**.  
Плотность должна быть не визуальной, а смысловой.

---

# 0. Главная формула системы

Интуитивность интерфейса строится из 6 вещей:

```text
Интуитивность =
понятная структура сайта
+ знакомые паттерны
+ визуальная иерархия
+ короткий текст
+ мгновенная обратная связь
+ доступность
```

Информационная плотность:

```text
Хорошая плотность = много смысла / мало шума
Плохая плотность = много элементов / мало понимания
```

# 1. Приоритеты интерфейса

Все элементы дели на 4 уровня:

## P0 — критическое
То, без чего пользователь не понимает экран.

Примеры:

- где я нахожусь;
- что это за страница;
- главное действие;
- ключевая информация;
- ошибка, если она есть.

**P0 всегда видно сразу.**

---

## P1 — частое
То, что используется часто.

Примеры:

- фильтры;
- сортировка;
- поиск;
- основные карточки;
- список объектов.

**P1 видно без глубокого поиска.**

---

## P2 — дополнительное
То, что нужно иногда.

Примеры:

- расширенные настройки;
- дополнительные поля;
- вторичные действия;
- справка.

**P2 можно прятать во вкладки, аккордеоны, drawer.**

---

## P3 — редкое
То, что почти никогда не нужно.

Примеры:

- опасные действия;
- архив;
- технические настройки;
- админские функции.

**P3 убирается глубже, но не исчезает полностью.**

# 2. Конкретные дизайн-токены

Если нет брендбука, используй эти значения по умолчанию.

---

## 2.1. Единицы

Используем:

```text
1 rem = 16px
Базовый шаг = 4px
Основная сетка = 8px
```

Почему так:

- 4px удобно для мелких внутренних расстояний;
- 8px — индустриальный стандарт для UI;
- почти все современные дизайн-системы используют 4/8px grid.

# 3. Отступы

## Базовая шкала spacing

| Токен | px | Где использовать |
|---|---:|---|
| `space-0` | 0 | Без отступа |
| `space-0-5` | 2 | Микро-коррекция, линии, focus offset |
| `space-1` | 4 | Очень маленькие зазоры, иконка внутри элемента |
| `space-2` | 8 | Иконка + текст, элементы внутри кнопки |
| `space-3` | 12 | Плотные строки, компактные блоки |
| `space-4` | 16 | Базовый padding карточки, mobile padding |
| `space-5` | 20 | Средний внутренний padding |
| `space-6` | 24 | Группы, секции, desktop padding |
| `space-8` | 32 | Крупные блоки, отступ между секциями |
| `space-10` | 40 | Большой вертикальный отступ |
| `space-12` | 48 | Разделы страницы |
| `space-16` | 64 | Большие секции, hero |
| `space-20` | 80 | Очень крупные разделы |
| `space-24` | 96 | Landing/marketing spacing |
| `space-32` | 128 | Редко, большие промо-блоки |

---

## Правила отступов

1. Между иконкой и текстом: **8px**.  
2. Между label и input: **6–8px**.  
3. Padding обычной кнопки: **12px 16px**.  
4. Padding карточки: **16px** compact, **24px** default.  
5. Между карточками: **16px** compact, **24px** default.  
6. Между смысловыми секциями: **32–48px**.  
7. Между заголовком и текстом: **8–12px**.  
8. Между заголовком секции и контентом: **16–24px**.  
9. Внутри группы расстояние всегда меньше, чем между группами.  
10. Не использовать случайные значения типа 13px, 17px, 29px.

# 4. Сетка и размеры экранов

## Важно про пиксели

В вебе мы проектируем не под физические пиксели экрана, а под **CSS pixels**.

Например:

- экран телефона может иметь физически 1170px ширины;
- но в CSS он может быть 390px;
- поэтому дизайнеру и разработчику важнее viewport: `390px`, `768px`, `1440px`.

---

## Основные breakpoint’ы

| Название | Ширина | Значение |
|---|---:|---|
| `xs` | 320–359px | Минимальный телефон |
| `sm` | 360px | Обычный маленький телефон |
| `md` | 768px | Планшет |
| `lg` | 1024px | Маленький ноутбук |
| `xl` | 1280px | Десктоп |
| `2xl` | 1536px | Большой десктоп |
| `wide` | 1920px+ | Широкие экраны |

---

## Под какие размеры обязательно проверять

Минимум проверять:

```text
320 × 568
360 × 800
390 × 844
768 × 1024
1024 × 768
1366 × 768
1440 × 900
1920 × 1080
```

---

## Контейнеры

| Экран | Padding страницы | Max-width |
|---|---:|---:|
| Mobile | 16px | 100% |
| Tablet | 24px | 100% |
| Desktop | 24–32px | 1200px |
| Data-heavy UI | 24–32px | 1280–1440px |
| Marketing page | 24–32px | 1120–1200px |

---

## Сетка

### Mobile

```text
4 колонки
gutter: 16px
margin: 16px
```

### Tablet

```text
8 колонок
gutter: 20px
margin: 24px
```

### Desktop

```text
12 колонок
gutter: 24px
margin: 32px
```

# 5. Типографика

## Шрифт

Если нет бренда:

```text
Inter
```

Fallback:

```css
font-family: Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

Почему Inter:

- хорошо читается в интерфейсах;
- поддерживает кириллицу;
- имеет много начертаний;
- часто используется в SaaS, dashboard, admin UI.

---

## Типографическая шкала

| Название | Desktop | Mobile | Weight | Где использовать |
|---|---:|---:|---:|---|
| Display | 56/64 | 40/48 | 700 | Главные промо-заголовки |
| H1 | 40/48 | 32/40 | 700 | Заголовок страницы |
| H2 | 32/40 | 28/36 | 700 | Большие секции |
| H3 | 24/32 | 22/30 | 600 | Подсекции |
| H4 | 20/28 | 20/28 | 600 | Заголовки карточек |
| Body L | 18/28 | 18/28 | 400 | Длинный текст |
| Body | 16/24 | 16/24 | 400 | Основной текст |
| UI | 14/20 | 14/20 | 400/500 | Интерфейс, таблицы |
| Caption | 12/16 | 12/16 | 400/500 | Подписи, метаданные |
| Tiny | 11/14 | не желательно | 500 | Только бейджи, редко |

Формат:  
`16/24` значит `font-size: 16px`, `line-height: 24px`.

---

## Правила текста

1. Основной читаемый текст: **16px минимум**.  
2. UI-текст в таблицах можно **14px**, но не меньше.  
3. Caption **12px** только для вторичной информации.  
4. Длина строки: **50–75 символов**.  
5. Для длинного текста max-width: **680–760px**.  
6. Не использовать `font-weight: 300` для интерфейса.  
7. Основные веса: **400, 500, 600, 700**.  
8. Заголовки должны быть короче 60 символов.  
9. Кнопки — 1–3 слова.  
10. Не использовать капслок для длинных фраз.

## Основной цвет

| Токен | HEX |
|---|---|

## Семантические цвета

| Смысл | HEX | Где использовать |
|---|---|---|
| Success | `#16A34A` | Успех, выполнено |
| Success bg | `#DCFCE7` | Фон success-сообщений |
| Warning | `#D97706` | Предупреждение |
| Warning bg | `#FEF3C7` | Фон warning |
| Danger | `#DC2626` | Ошибка, удаление |
| Danger bg | `#FEE2E2` | Фон error |
| Info | `#0284C7` | Информация |
| Info bg | `#E0F2FE` | Фон info |

---

## Правила цвета

1. Не передавать смысл только цветом.  
   Ошибка должна иметь цвет + иконку + текст.

2. Красный использовать только для:
   - ошибок;
   - удаления;
   - критических состояний.

3. Зелёный использовать только для:
   - успеха;
   - подтверждения;
   - положительного статуса.

4. Primary color нельзя использовать для всего подряд.  
   Он должен означать: “сюда нужно нажать”.

5. Фон интерфейса:
   - page background: `neutral-50`;
   - cards/surfaces: `white`;
   - border: `neutral-200`.

6. Контраст текста:
   - обычный текст минимум **4.5:1**;
   - крупный текст минимум **3:1**;
   - иконки/границы минимум **3:1**.

# 8. Motion / анимации

## Длительности

| Тип | Время |
|---|---:|
| Hover/focus | 100–150ms |
| Small UI transition | 150–200ms |
| Dropdown/modal | 200–250ms |
| Page transition | 250–350ms |
| Loading skeleton | пока грузится |

---

## Easing

Default:

```css
cubic-bezier(0.2, 0, 0, 1)
```

Для выхода:

```css
cubic-bezier(0.4, 0, 1, 1)
```

Правила:

1. Анимация помогает понять изменение, а не украшает.  
2. Не анимировать всё подряд.  
3. Уважать `prefers-reduced-motion`.  
4. Если действие мгновенное — не делать длинную анимацию.  
5. Если ожидание больше 300–500ms — показать feedback.

# 9. Компоненты

---

## 9.1. Кнопки

### Размеры

| Размер | Height | Padding | Font |
|---|---:|---:|---:|
| Small | 32px | 8px 12px | 14px |
| Medium | 40px | 10px 16px | 14px |
| Large | 48px | 12px 20px | 16px |

Default:

```text
height: 40px
radius: 8px
font-weight: 500/600
gap icon-text: 8px
```

---

### Типы кнопок

1. `Primary` — главное действие.
2. `Secondary` — вторичное действие.
3. `Ghost` — тихое действие.
4. `Danger` — удаление/опасность.
5. `Icon button` — только иконка.
6. `Link button` — действие выглядит как ссылка.

---

### Правила кнопок

1. На экране максимум **1 primary action**.  
2. Primary button ставится справа в форме/modal или сверху справа в page header.  
3. Danger button не должен быть рядом с primary без визуального разделения.  
4. Disabled button должен иметь объяснение, почему он disabled.  
5. Icon-only button всегда имеет tooltip/aria-label.  
6. Минимальная click/tap зона: **44×44px**.  
   Даже если визуально кнопка 32px, hit area должна быть больше.

---

## 9.2. Inputs

### Размеры

| Размер | Height | Padding |
|---|---:|---:|
| Compact | 36px | 8px 12px |
| Default | 40px | 10px 12px |
| Large | 48px | 12px 14px |

Default:

```text
height: 40px
radius: 8px
border: #CBD5E1
focus: #2563EB
error: #DC2626
```

---

### Поле должно иметь

- label;
- placeholder только как пример, не как label;
- help text, если поле сложное;
- error message, если ошибка;
- success state, если нужно;
- disabled/read-only state.

---

### Ошибка поля

Плохой пример:

```text
Ошибка
```

Хороший пример:

```text
Введите email в формате name@example.com
```

Формула ошибки:

```text
Что не так + как исправить
```

---

## 9.3. Forms

Правила:

1. Label сверху, не внутри поля.  
2. Один главный action в конце формы.  
3. Ошибки показывать рядом с полем.  
4. Валидация:
   - во время ввода — осторожно;
   - после blur — хорошо;
   - после submit — обязательно.
5. Не делать форму в 3 колонки.  
6. На mobile всегда 1 колонка.  
7. На desktop можно 2 колонки, если поля короткие.  
8. Обязательные поля отмечать явно.  
9. Не прятать важные поля в placeholder.  
10. После успешного действия показать результат.

---

## 9.4. Tables

Для информационно-плотного UI таблицы очень важны.

### Размеры

| Элемент | Размер |
|---|---:|
| Header row | 40px |
| Default row | 44px |
| Compact row | 36px |
| Cell padding | 8px 12px |
| Font | 14px |
| Border | 1px `neutral-200` |

---

### Правила таблиц

1. Текст слева.  
2. Числа справа.  
3. Даты справа или слева, но единообразно.  
4. Статусы — badge.  
5. Действия — справа.  
6. Header sticky, если таблица длинная.  
7. Первая важная колонка может быть sticky.  
8. Сортировка видна и понятна.  
9. Фильтры над таблицей.  
10. Empty state вместо пустого белого пространства.  
11. Pagination или infinite scroll — не смешивать без причины.  
12. В таблице не делать много ярких цветов.  
13. Long text truncate + tooltip/details.  
14. Для массовых действий использовать checkbox selection.  
15. Для плотности использовать compact mode только на desktop.

---

## 9.5. Cards

Default card:

```text
background: white
border: 1px solid #E2E8F0
radius: 12px
padding: 16px или 24px
gap: 12px или 16px
```

Правила:

1. В карточке один главный смысл.  
2. Заголовок сверху.  
3. Действия справа сверху или внизу.  
4. Не помещать в карточку всё подряд.  
5. Карточки одного типа должны быть одинаковой структуры.  
6. Если карточек много и они однотипные — возможно нужна таблица.

---

## 9.6. Navigation

### Header

```text
Desktop height: 64px
Mobile height: 56px
```

Содержит:

- logo;
- primary navigation;
- search, если сайт большой;
- user/account actions;
- mobile menu button.

---

### Sidebar

Для сложных сайтов/app:

```text
width: 280px
collapsed: 72px
nav item height: 40px
icon: 20px
gap icon-text: 12px
```

Правила:

1. Активный раздел подсвечен.  
2. Иконка без текста допустима только в collapsed mode.  
3. Вложенность не глубже 2 уровней в sidebar.  
4. Если уровней больше — нужны отдельные страницы-разделы.  
5. Sidebar не должен конкурировать с главным контентом.

---

## 9.7. Breadcrumbs

Использовать, если глубина больше 2 уровней.

Формат:

```text
Главная › Раздел › Подраздел › Текущая страница
```

Правила:

1. Последний элемент не ссылка.  
2. Названия совпадают с заголовками страниц.  
3. Breadcrumbs не заменяют главное меню.  
4. На mobile можно сокращать середину.

---

## 9.8. Modals

Размеры:

| Тип | Width |
|---|---:|
| Small | 400px |
| Medium | 560px |
| Large | 720px |
| XL | 960px |

Правила:

1. Modal — для короткой задачи.  
2. Длинные формы лучше делать отдельной страницей или drawer.  
3. Заголовок обязателен.  
4. Primary action справа снизу.  
5. Cancel слева от primary.  
6. Escape закрывает, если нет потери данных.  
7. При несохранённых данных предупреждать.  
8. Не открывать modal поверх modal.

---

## 9.9. Toast / уведомления

Default:

```text
width: до 420px
position: top-right desktop
position: bottom mobile
duration: 4–6 seconds
```

Правила:

1. Toast не должен содержать критически важную информацию, которая исчезнет навсегда.  
2. Success toast короткий.  
3. Error toast должен иметь действие или путь исправления.  
4. Если действие важное — использовать inline alert, не toast.  
5. Не показывать больше 3 toast одновременно.

# 10. Информационная архитектура сайта

## Главная структура

Для большинства сайтов используй такую модель:

```text
Главная
├── Раздел / Категория
│   ├── Обзор раздела
│   ├── Список объектов
│   ├── Детальная страница
│   └── Действие / форма
├── Второй раздел
├── Кейсы / примеры
├── База знаний / помощь
├── О компании / доверие
└── Контакты / действие
```

Для app/dashboard:

```text
Dashboard
├── Основные сущности
│   ├── List
│   ├── Detail
│   ├── Create/Edit
│   └── Settings
├── Analytics
├── Reports
├── Users / Team
├── Billing
├── Integrations
├── Settings
└── Help
```

---

## Правила дерева сайта

1. Главные разделы: **5–7 пунктов**.  
2. Глубина для 80% задач: максимум **3 клика**.  
3. Каждый раздел отвечает на один пользовательский вопрос.  
4. Названия разделов — словами пользователя, не внутренним жаргоном компании.  
5. Не смешивать разные принципы группировки.  
   Например, нельзя одновременно группировать по продуктам, аудиториям и отделам.  
6. В каждом разделе должна быть обзорная страница.  
7. На глубоких страницах нужны breadcrumbs.  
8. Поиск обязателен, если страниц/объектов много.  
9. Не должно быть тупиковых страниц.  
10. В конце страницы всегда есть следующий логичный шаг.

# 11. Шаблон каждой страницы

Каждая страница должна иметь:

```text
1. Page title
2. Краткое описание / контекст
3. Primary action
4. Основной контент
5. Secondary actions
6. Состояния: loading, empty, error, success
7. Навигационный путь назад/вверх
```

## Page header

Desktop:

```text
Title слева
Description под title
Primary action справа
Secondary actions рядом или в menu
```

Mobile:

```text
Title сверху
Action ниже или sticky bottom
```

```text
Disabled
Loading
Empty
Error
```

# 13. 100 законов для этой дизайн-системы

Это не мистические «законы», а практические правила из UX, психологии, биологии, физики восприятия, инженерии и информационной архитектуры.

---

## Физика экрана и восприятия

**1. Закон CSS-пикселя**  
Проектируй в CSS px, а не в физических пикселях устройства.

**2. Закон viewport’ов**  
Проверяй дизайн минимум на 320, 390, 768, 1366, 1440 и 1920px.

**3. Закон визуального угла**  
Чем дальше экран от глаз, тем крупнее должны быть элементы. Для desktop основной текст 16px, для плотных UI-элементов минимум 14px.

**4. Закон контраста**  
Текст должен иметь контраст минимум 4.5:1, иначе мозг тратит больше энергии на чтение.

**5. Закон тонких линий**  
Не используй светлый текст тонким весом. `font-weight: 300` + серый цвет = плохая читаемость.

**6. Закон антишума**  
Чем больше границ, теней и цветов, тем ниже сигнал/шум. Убирай всё, что не помогает задаче.

**7. Закон стабильной сетки**  
Глаз быстрее считывает интерфейс, если элементы стоят по повторяющимся линиям.

**8. Закон размера цели**  
Чем меньше кнопка, тем дольше и сложнее в неё попасть. Минимальная зона клика — 44×44px.

**9. Закон расстояния до цели**  
Частые действия должны быть ближе к месту принятия решения.

**10. Закон края экрана**  
Края и углы экрана легче использовать как опорные зоны, особенно в desktop app.

---

## Биология зрения и моторики

**11. Закон фовеального зрения**  
Человек видит детали только в маленькой центральной зоне. Не разбрасывай связанную информацию далеко друг от друга.

**12. Закон периферии**  
Периферия хорошо замечает движение и контраст, но плохо читает текст. Поэтому уведомления можно показывать сбоку, но текст должен быть коротким.

**13. Закон цветовых ограничений**  
Не все люди различают красный/зелёный. Статус должен иметь цвет + текст + иконку.

**14. Закон утомления глаз**  
Длинный текст на ярком белом фоне без воздуха утомляет. Ограничивай ширину текста и line-height.

**15. Закон мелкой моторики**  
На touch-устройствах мелкие элементы вызывают ошибки. Mobile controls — 44–48px.

**16. Закон большого пальца**  
На mobile важные действия лучше располагать ближе к нижней части экрана, но не перекрывать системные жесты.

**17. Закон биологического ритма внимания**  
Человек не может долго удерживать внимание на равномерной массе информации. Нужны заголовки, группы, паузы.

**18. Закон усталости от выбора**  
Каждый лишний вариант увеличивает нагрузку. Показывай сначала частые варианты, редкие — в “More”.

**19. Закон мгновенной обратной связи**  
Мозг ждёт реакции сразу. Если после клика ничего не изменилось — пользователь думает, что система сломалась.

**20. Закон избегания боли**  
Красные ошибки, резкие модалки и блокировки воспринимаются как стресс. Ошибка должна помогать, а не пугать.

---

## Психология и когнитивная нагрузка

**21. Закон “не заставляй думать”**  
Если человек спрашивает “куда нажать?”, интерфейс недостаточно ясен.

**22. Закон узнавания вместо памяти**  
Показывай варианты. Не заставляй вспоминать команды, названия или пути.

**23. Закон Хика**  
Чем больше вариантов выбора, тем дольше решение. Группируй и сокращай.

**24. Закон 4±1 чанков**  
Рабочая память ограничена. В одном блоке лучше 3–5 ключевых элементов.

**25. Закон когнитивной нагрузки**  
Если пользователь одновременно должен читать, искать, сравнивать и помнить — экран перегружен.

**26. Закон одного главного действия**  
На одном экране один primary action. Всё остальное визуально тише.

**27. Закон последовательного раскрытия**  
Сначала суть, потом детали. Не показывай advanced-настройки новичку сразу.

**28. Закон предсказуемости**  
Похожие элементы должны вести себя одинаково.

**29. Закон привычных паттернов**  
Не изобретай новый select, checkbox, tabs и navigation без крайней необходимости.

**30. Закон ментальной модели**  
Структура сайта должна совпадать с тем, как пользователь думает о задаче.

**31. Закон эффекта первого экрана**  
Первый экран должен сразу объяснить: где я, что здесь есть, что делать дальше.

**32. Закон серийной позиции**  
Люди лучше запоминают начало и конец списка. Важные пункты ставь туда.

**33. Закон дефолтов**  
Хорошие значения по умолчанию уменьшают количество решений.

**34. Закон потери**  
Пользователи сильнее боятся потерять, чем хотят получить. Опасные действия должны быть ясно обозначены.

**35. Закон доверия**  
Непоследовательный UI снижает доверие даже к хорошему продукту.

---

## Гештальт и визуальная организация

**36. Закон близости**  
Связанные элементы ближе друг к другу, чем к другим элементам.

**37. Закон подобия**  
Одинаковые элементы выглядят одинаково.

**38. Закон общей области**  
Элементы внутри одной карточки воспринимаются как группа.

**39. Закон непрерывности**  
Глаз идёт по линиям. Используй выравнивание.

**40. Закон фигуры и фона**  
Главный объект должен ясно отделяться от фона.

**41. Закон иерархии**  
Размер, вес, цвет и позиция показывают важность.

**42. Закон визуального веса**  
Крупный, тёмный, насыщенный элемент воспринимается важнее.

**43. Закон пустого пространства**  
Whitespace — это не пустота, а инструмент группировки.

**44. Закон ритма**  
Повторяющиеся размеры и отступы создают ощущение порядка.

**45. Закон якорей**  
Пользователь ищет стабильные точки: header, sidebar, breadcrumbs, title.

---

## Информационная архитектура

**46. Закон карты сайта**  
Нельзя качественно проектировать экраны без дерева сайта.

**47. Закон 5–7 разделов**  
Главное меню лучше держать в диапазоне 5–7 пунктов.

**48. Закон 3 кликов**  
Ключевые задачи должны быть достижимы за 1–3 шага.

**49. Закон “ты здесь”**  
Пользователь всегда должен понимать своё положение в системе.

**50. Закон хлебных крошек**  
Если глубина больше 2 уровней — нужны breadcrumbs.

**51. Закон названий пользователя**  
Разделы называются языком пользователя, не языком компании.

**52. Закон отсутствия тупиков**  
Каждая страница предлагает следующий шаг.

**53. Закон поиска как страховки**  
Если пользователь не понял структуру, поиск должен его спасти.

**54. Закон одинаковых названий**  
Один объект называется одинаково во всём интерфейсе.

**55. Закон информационного запаха**  
Ссылка должна ясно обещать, что будет после клика.

---

## Контент и язык

**56. Закон коротких фраз**  
Чем короче фраза, тем быстрее она сканируется.

**57. Закон глагола в кнопке**  
Кнопка должна отвечать на вопрос “что сделать?”: “Сохранить”, “Создать”, “Отправить”.

**58. Закон конкретики**  
“Ошибка сохранения” хуже, чем “Не удалось сохранить: нет подключения”.

**59. Закон полезного заголовка**  
Заголовок должен объяснять смысл блока, а не быть декоративным.

**60. Закон сканирования**  
Люди сначала сканируют, потом читают. Используй списки, подзаголовки, выделения.

**61. Закон одного термина**  
Не называй одно и то же “заказ”, “заявка”, “запрос”, если это один объект.

**62. Закон без канцелярита**  
“Осуществить отправку данных” → “Отправить данные”.

**63. Закон чисел**  
Большие числа форматируй: `1 200 000` → `1,2 млн`, если точность не важна.

**64. Закон дат**  
Даты должны быть единообразны: `12.06.2026`, `12 июня 2026`, но не смесь.

**65. Закон пустого состояния**  
Empty state должен объяснять, что здесь будет и как начать.

---

## Цвет

**66. Закон семантики цвета**  
Цвет должен иметь смысл, а не просто украшать.

**67. Закон акцентной экономии**  
Если всё синее — ничего не синее. Акцент только для важного.

**68. Закон красного цвета**  
Красный — только ошибка, опасность, удаление.

**69. Закон фона**  
Контент лучше читать на спокойном фоне: white, neutral-50, neutral-100.

**70. Закон тёмной темы**  
Тёмная тема — не инверсия. Не используй чистый `#000` как основной фон.

---

## Типографика

**71. Закон читаемого body**  
Основной текст 16px/24px.

**72. Закон плотного UI**  
Для таблиц и control labels допустимо 14px/20px.

**73. Закон line-height**  
Для текста line-height 1.4–1.6, для UI 1.2–1.4.

**74. Закон длины строки**  
Длинный текст — 50–75 символов в строке.

**75. Закон табличных цифр**  
В колонках с числами используй tabular numbers.

---

## Layout и плотность

**76. Закон 8px-сетки**  
Все основные расстояния кратны 8px, мелкие — 4px.

**77. Закон группировки**  
Внутри группы 8–16px, между группами 24–48px.

**78. Закон верхнего приоритета**  
Самое важное — выше и левее в desktop-интерфейсе.

**79. Закон адаптации**  
На mobile не сжимай desktop, а перестраивай layout.

**80. Закон прогрессивной плотности**  
Новичку — проще, опытному пользователю — compact mode.

---

## Взаимодействие

**81. Закон состояния системы**  
После действия пользователь должен видеть результат.

**82. Закон загрузки**  
Если загрузка больше 300–500ms, показывай skeleton/spinner.

**83. Закон undo**  
Лучше дать “Отменить”, чем спрашивать подтверждение на каждое действие.

**84. Закон опасного действия**  
Удаление требует подтверждения или возможности восстановления.

**85. Закон disabled-состояния**  
Если кнопка disabled, пользователь должен понимать почему.

**86. Закон keyboard-first**  
Всё должно работать с клавиатуры.

**87. Закон focus ring**  
Focus должен быть видимым, не удалять outline без замены.

**88. Закон hover не равен доступность**  
Информация не должна быть доступна только при hover.

**89. Закон mobile gestures**  
Жесты должны иметь видимую альтернативу.

**90. Закон не ломать Back**  
Кнопка браузера “назад” должна вести ожидаемо.

---

## Доступность и инженерия

**91. Закон WCAG AA**  
Минимальная цель — WCAG 2.2 AA.

**92. Закон семантики HTML**  
Кнопка — `<button>`, ссылка — `<a>`, а не div с onClick.

**93. Закон alt-текста**  
Смысловые изображения имеют alt, декоративные — пустой alt.

**94. Закон масштабирования**  
Интерфейс должен работать при zoom 200%.

**95. Закон reduced motion**  
Если пользователь отключил анимации — уважать это.

**96. Закон скорости**  
Интерфейс должен реагировать до 100ms, если возможно.

**97. Закон Core Web Vitals**  
LCP желательно до 2.5s, CLS до 0.1, INP до 200ms.

**98. Закон ошибок разработки**  
Каждый компонент должен иметь loading, error, empty state.

**99. Закон документации**  
Если компонент не описан — он будет использован неправильно.

**100. Закон тестирования**  
Проверяй интерфейс на реальных людях. Даже 5 пользователей часто находят основные проблемы.

# 15. Алгоритм проектирования сайта

Делай в таком порядке:

## Шаг 1. Определи топ-задачи

Выпиши 5–10 главных задач пользователя.

# 18. Короткая финальная формулировка

Твой стиль можно описать так:

> **Информационно-плотный функциональный минимализм, построенный на сильной информационной архитектуре, прогрессивном раскрытии и дизайн-токенах.**

Или короче:

> **IA-first Dense UX Design System.**

Это подход для сайтов, SaaS, dashboard, marketplaces, каталогов, образовательных платформ и любых сложных интерфейсов, где важно уместить много информации, но не перегрузить человека. 
# Backend Design System v1.0  
## «Чистая библиотека»

Главная цель:

> **Backend должен выглядеть не как папка с хаотичными controller/service/utils, а как хорошо собранная библиотека возможностей.**

То есть разработчик открывает проект и быстро понимает:

```text
где API
где бизнес-логика
где база
где кэш
где контракты
где ошибки
где тесты
как добавить новую функцию
как не сломать старую
```

# 0. Главная формула backend’а

```text
Понятный backend =
Domain Tree
+ API Tree
+ Request Tree
+ Data Access Rules
+ Resource Budgets
+ Contracts
+ Observability
```

Ресурсная эффективность:

```text
Хорошая эффективность = нужные данные / минимум запросов, CPU, RAM, IO, payload

Плохая эффективность = много endpoint’ов + N+1 + лишние поля + тяжёлые JOIN + нет кэша
```

Главное правило:

> **Каждый backend-запрос должен иметь понятную причину существования.**

# 1. Какой стиль архитектуры брать по умолчанию

Для большинства проектов лучший старт:

> **Modular Monolith + Clean/Hexagonal Architecture + API-first + CQRS-lite + BFF/View API при необходимости.**

Расшифровка:

## 1.1. Modular Monolith

Не микросервисы сразу, а один backend, но внутри разделённый на модули.

```text
modules/
├── users/
├── projects/
├── tasks/
├── billing/
├── notifications/
└── reports/
```

Почему так хорошо:

- меньше DevOps-сложности;
- проще транзакции;
- меньше сетевых вызовов;
- легче дебажить;
- можно потом вынести модуль в сервис.

---

## 1.2. Clean / Hexagonal Architecture

Внутри каждого модуля слои:

```text
api -> application -> domain <- infrastructure
```

Где:

- `api` — HTTP, controllers, routes, schemas;
- `application` — use cases, commands, queries;
- `domain` — бизнес-правила;
- `infrastructure` — база, Redis, внешние API, очереди.

Главное:

> Бизнес-логика не должна жить в controller’ах.

---

## 1.3. CQRS-lite

Не обязательно делать сложный CQRS, но полезно разделять:

```text
Commands = изменить состояние
Queries = прочитать данные
```

Примеры:

```text
CreateProjectCommand
UpdateTaskStatusCommand

GetProjectDetailQuery
ListProjectTasksQuery
```

Это сразу делает backend понятнее.

---

## 1.4. BFF / View API

Если экрану нужно 5–10 разных запросов, лучше сделать один аккуратный агрегирующий endpoint:

```text
GET /api/v1/views/project-detail/{projectId}
```

Но важно:

> View API не должен становиться мусорным mega-controller’ом.  
> Он только собирает данные из нормальных domain-use-cases.

# 2. Приоритеты данных: B0 / B1 / B2 / B3

Как в UI у тебя есть P0/P1/P2/P3, в backend делаем так.

---

## B0 — критические данные

Без них экран или действие невозможны.

Примеры:

- текущий пользователь;
- права доступа;
- основная сущность страницы;
- статус объекта;
- обязательные поля формы.

Правило:

> B0 грузится первым, быстро, стабильно, без тяжёлых вычислений.

---

## B1 — частые данные

Нужны почти всегда, но не всегда блокируют экран.

Примеры:

- список последних объектов;
- краткая статистика;
- owner summary;
- счётчики;
- статусы.

Правило:

> B1 можно включать через `include`, кэшировать, batch’ить.

---

## B2 — дополнительные данные

Нужны иногда.

Примеры:

- активность;
- история;
- комментарии;
- расширенные настройки;
- вторичные табы.

Правило:

> B2 грузится lazy: по вкладке, раскрытию, drawer’у.

---

## B3 — редкие / тяжёлые данные

Примеры:

- audit log;
- export;
- отчёты;
- архив;
- админские операции.

Правило:

> B3 уходит в отдельный endpoint, background job или export.

# 5. Правила API paths

## 5.1. Используй существительные

Хорошо:

```text
GET /projects
GET /projects/{projectId}
GET /projects/{projectId}/tasks
```

Плохо:

```text
GET /getProjects
POST /createNewProject
GET /projectTasksList
```

---

## 5.2. Версия API обязательна

```text
/api/v1/projects
```

---

## 5.3. Вложенность максимум 2 уровня

Хорошо:

```text
GET /projects/{projectId}/tasks
```

Плохо:

```text
GET /organizations/{orgId}/projects/{projectId}/tasks/{taskId}/comments/{commentId}/likes
```

Если вложенность глубокая — используй фильтры:

```text
GET /comments?taskId=123
GET /likes?commentId=456
```

---

## 5.4. Actions отдельно и редко

Для нестандартных действий:

```text
POST /projects/{projectId}:archive
POST /tasks/{taskId}:complete
POST /reports/{reportId}:export
```

Не превращать всё в actions.

---

## 5.5. List endpoint всегда имеет фильтры, сортировку и лимит

```text
GET /tasks?projectId=123&status=open&sort=-createdAt&limit=50&cursor=abc
```

Обязательные параметры для списков:

```text
limit
cursor/page
sort
filters
```

Нельзя делать:

```text
GET /tasks
```

и возвращать 50 000 записей.

# 6. Request Tree — сердце системы

Главная идея:

> Для каждого экрана заранее описывается дерево backend-запросов: что нужно сразу, что можно позже, что кэшируется, что batch’ится.

Пример:

```text
GET /api/v1/views/project-detail/{projectId}?sections=main,tasks,stats
```

Request tree:

```text
project-detail
├── main [B0]
│   ├── project.summary
│   ├── currentUser.permissions
│   └── owner.summary
│
├── tasks [B1]
│   ├── tasks.list limit=20
│   └── users.batch-get assignees
│
├── stats [B1]
│   └── project.stats.readModel
│
├── activity [B2 lazy]
│   └── activity.list limit=20
│
└── audit [B3 admin lazy]
    └── audit.list
```

---

## 6.1. Формат request contract

Можно хранить такие контракты в `contracts/views/`.

```yaml
view: project-detail
route: GET /api/v1/views/project-detail/{projectId}

budget:
  p95Ms: 450
  maxPayloadKb: 150
  maxDbQueries: 8

sections:
  main:
    priority: B0
    required: true
    nodes:
      - id: project
        useCase: projects.getSummary
        fields: [id, name, status, ownerId, updatedAt]
        cache:
          ttlSec: 30

      - id: permissions
        useCase: projects.getCurrentUserPermissions
        cache:
          ttlSec: 15

      - id: owner
        useCase: users.getSummary
        dependsOn: [project.ownerId]
        fields: [id, name, avatarUrl]
        cache:
          ttlSec: 300

  tasks:
    priority: B1
    required: false
    nodes:
      - id: tasks
        useCase: tasks.listByProject
        params:
          limit: 20
          sort: -updatedAt

  activity:
    priority: B2
    lazy: true
    nodes:
      - id: activity
        useCase: activity.listByProject
        params:
          limit: 20
```

---

## 6.2. Как backend выполняет request tree

Алгоритм:

```text
1. Принять запрос
2. Проверить auth
3. Проверить права
4. Прочитать sections
5. Построить дерево зависимостей
6. Удалить ненужные ветки
7. Проверить cache
8. Сгруппировать одинаковые запросы
9. Выполнить независимые nodes параллельно
10. Ограничить concurrency
11. Собрать response
12. Записать metrics/logs
```

# 10. `fields` и `include`

Для экономии payload’а:

```text
GET /projects/123?fields=id,name,status
```

Для аккуратного добавления связанных данных:

```text
GET /projects/123?include=ownerSummary,stats
```

Правила:

1. `fields` — только whitelist.
2. `include` — только заранее разрешённые значения.
3. `include` возвращает summary, не огромную вложенную сущность.
4. Максимальная глубина include — 1.
5. Тяжёлые include запрещены, для них отдельный endpoint.
6. `include=all` запрещён.

# 13. Data access rules

## 13.1. Никакого `SELECT *`

Плохо:

```sql
SELECT * FROM projects;
```

Хорошо:

```sql
SELECT id, name, status, owner_id, updated_at
FROM projects
WHERE organization_id = ?
ORDER BY updated_at DESC
LIMIT 50;
```

---

## 13.2. Никакого N+1

Плохо:

```text
1 запрос: получить 50 задач
50 запросов: получить assignee для каждой задачи
```

Хорошо:

```text
1 запрос: получить задачи
1 batch-запрос: получить всех assignee по ids
```

---

## 13.3. Индекс под каждый частый фильтр

Если есть endpoint:

```text
GET /tasks?projectId=123&status=open&sort=-createdAt
```

Значит нужен индекс примерно под:

```text
project_id
status
created_at
```

---

## 13.4. Cursor pagination по умолчанию

Для больших списков лучше:

```text
limit + cursor
```

а не:

```text
limit + offset
```

Потому что `offset` на больших таблицах становится дорогим.

---

## 13.5. Count осторожно

Плохо:

```text
На каждый list endpoint делать точный COUNT(*) по огромной таблице.
```

Лучше:

```text
hasMore
approximateCount
cachedCount
separate stats endpoint
```

# 14. Resource budgets

Задай бюджеты по умолчанию.

| Метрика | Default |
|---|---:|
| P95 для простого read | до 200–300ms |
| P95 для сложного view | до 400–600ms |
| Write request | до 300–700ms |
| Heavy operation | 202 + background job |
| Max list limit | 100 |
| Default list limit | 20–50 |
| Max response P0 | 100–200KB |
| Slow DB query | 100–200ms |
| Max DB queries на request | 5–10 |
| Max request body | 512KB–2MB |
| Export/report | async job |

Это не закон физики, а стартовые ограничения.

Главная идея:

> Если endpoint не проходит budget — он не готов.

# 15. Caching strategy

Кэш должен быть не случайным, а системным.

## 15.1. Уровни кэша

```text
L0: request memoization
L1: application memory cache
L2: Redis/cache storage
L3: CDN/HTTP cache
L4: precomputed read models
```

---

## 15.2. Что кэшировать

Хорошо кэшируются:

```text
справочники
настройки
public content
summary DTO
права на короткий TTL
read models
статистика
```

Плохо кэшируются:

```text
часто меняющиеся деньги
критичные permission checks без versioning
персональные данные без user/tenant key
```

---

## 15.3. Cache key

Формат:

```text
{env}:{module}:{resource}:{id}:{version}:{fieldsHash}:{userOrTenantScope}
```

Пример:

```text
prod:projects:summary:prj_123:v4:fields_ab12:org_77
```

---

## 15.4. Правила кэша

1. Нет кэша без TTL.
2. Нет кэша без owner’а.
3. Нет кэша без стратегии invalidation.
4. Private cache key должен учитывать user/tenant/permissions.
5. Cache hit/miss логируется.
6. Кэш — не замена нормальным индексам.
7. Нельзя кэшировать мусорный тяжёлый response вместо исправления архитектуры.

---

# 16. Batching

Batch нужен, когда много однотипных запросов.

Плохо:

```text
GET /users/1
GET /users/2
GET /users/3
```

Хорошо:

```text
POST /users:batch-get
```

Body:

```json
{
  "ids": ["usr_1", "usr_2", "usr_3"],
  "fields": ["id", "name", "avatarUrl"]
}
```

Response:

```json
{
  "data": {
    "usr_1": {
      "id": "usr_1",
      "name": "Alex"
    },
    "usr_2": {
      "id": "usr_2",
      "name": "Maria"
    }
  }
}
```

Правила:

1. Batch endpoints нужны для внутренних/BFF сценариев.
2. Ограничь max ids, например 100.
3. Сохраняй порядок, если возвращаешь array.
4. Лучше map по id, если порядок не важен.
5. Batch должен поддерживать fields.
6. Batch не должен обходить permissions.

# 17. Async jobs

Если операция долгая — не держи HTTP request.

Плохо:

```text
POST /reports/export
ждать 40 секунд
```

Хорошо:

```text
POST /reports/{reportId}:export
```

Response:

```json
{
  "data": {
    "jobId": "job_123",
    "status": "queued"
  }
}
```

Потом:

```text
GET /jobs/job_123
```

Response:

```json
{
  "data": {
    "id": "job_123",
    "status": "done",
    "resultUrl": "https://cdn.example.com/export.csv"
  }
}
```

Правила:

1. Всё дольше 1–2 секунд — кандидат в job.
2. Job должен иметь status.
3. Job должен иметь retry policy.
4. Job должен быть idempotent.
5. Ошибка job должна быть читаемой.
6. Большие файлы — object storage/CDN, не через backend memory.

# 28. 50 senior-законов backend’а

## Архитектура

1. Backend должен быть понятен по дереву папок.  
2. Модуль должен выглядеть как библиотека.  
3. Controller не место для бизнес-логики.  
4. Domain не должен знать про HTTP.  
5. Infrastructure не должна диктовать бизнес-правила.  
6. Не начинай с микросервисов без причины.  
7. Modular monolith лучше хаотичного distributed mess.  
8. Границы модулей важнее количества файлов.  
9. Не делай abstraction ради abstraction.  
10. Публичный contract важнее внутренней реализации.

## API

11. API path должен читаться как карта продукта.  
12. Endpoint без владельца превращается в мусор.  
13. List endpoint без pagination — баг.  
14. `include=all` — почти всегда ошибка.  
15. DTO не должен быть копией DB entity.  
16. Ошибки должны иметь стабильные codes.  
17. Версионирование API нужно до первой большой поломки.  
18. Один термин — один смысл во всём backend’е.  
19. Не смешивай admin API и user API без явной границы.  
20. Public API должен быть документирован.

## Request tree

21. Экран должен иметь request tree.  
22. B0 данные грузятся первыми.  
23. B2/B3 данные не должны тормозить B0.  
24. Если экран требует 15 запросов — нужен BFF/View API.  
25. Одинаковые запросы должны dedupe’иться.  
26. Независимые nodes можно выполнять параллельно.  
27. Зависимые nodes должны быть явно описаны.  
28. У каждого node должен быть timeout.  
29. У каждого view должен быть payload budget.  
30. Partial response лучше полного падения из-за B2 блока.

## Performance

31. Сначала убери N+1, потом добавляй кэш.  
32. Кэш не лечит плохую модель данных.  
33. `SELECT *` — враг больших систем.  
34. Индекс должен следовать за частым фильтром.  
35. Большие exports должны быть async.  
36. Большие файлы не должны проходить через RAM backend’а.  
37. Count на больших таблицах должен быть осознанным.  
38. Payload — это тоже ресурс.  
39. Slow query должен быть виден в логах.  
40. Performance budget должен быть частью DoD.

## Data

41. Транзакции должны быть короткими.  
42. Миграции должны быть обратимыми или безопасными.  
43. Схема базы — часть архитектуры, а не побочный эффект ORM.  
44. Soft delete должен быть осознанным, не автоматическим мусором.  
45. Audit log нужен для опасных действий.  
46. Read model лучше тяжёлого JOIN на каждом запросе.  
47. Денормализация допустима, если она документирована.  
48. Data ownership должен быть понятен.  
49. Multi-tenant фильтр нельзя доверять frontend’у.  
50. Backup/restore — часть backend-системы.

# Database Design System v1.0  
## «Чистое хранилище»

Главная цель:

> **База данных должна быть не местом, куда backend “просто что-то сохраняет”, а строгой библиотекой фактов, связей и быстрых путей доступа.**

---

# 0. Главная формула DB

```text
Понятная база =
Domain Data Tree
+ Schema Tree
+ Access Pattern Tree
+ Index Tree
+ Migration Tree
+ Data Lifecycle
+ Observability
```

Эффективность базы:

```text
Хорошая эффективность =
минимум строк прочитано
+ минимум колонок выбрано
+ правильный индекс
+ короткая транзакция
+ маленький payload
+ предсказуемый query plan
```

Плохая эффективность:

```text
SELECT *
+ N+1
+ нет индексов
+ OFFSET на миллионах строк
+ тяжёлый COUNT(*)
+ случайные JSON-поля
+ огромные JOIN без нужды
+ бизнес-логика спрятана в хаосе
```

# 1. Какой подход брать по умолчанию

Для большинства SaaS/dashboard/marketplace/backend-систем:

> **PostgreSQL / SQL-first + нормализованная write-model + денормализованные read-models там, где нужно.**

Правило senior-разработчика:

> **Таблицы проектируются от домена.  
> Индексы проектируются от запросов.  
> Read models проектируются от экранов.**

То есть:

```text
Domain → таблицы
Request Tree → access patterns
Access Patterns → индексы
Тяжёлые экраны → read models
```


# 4. D0 / D1 / D2 / D3 приоритеты данных

Как в UI были P0/P1/P2/P3, в backend B0/B1/B2/B3, для DB делаем:

---

## D0 — критические данные

Без них нельзя выполнить действие.

Примеры:

```text
user
organization membership
project core row
permissions
payment state
task status
```

Правила:

```text
D0 должен быть быстрым
D0 должен иметь строгие constraints
D0 не должен зависеть от тяжёлых JOIN
D0 не должен лежать в мутном JSON
```

---

## D1 — частые данные

Нужны почти всегда.

Примеры:

```text
project list item
task list item
short user profile
stats summary
status counters
```

Правила:

```text
D1 оптимизируется индексами
D1 можно хранить в read model
D1 должен возвращать только нужные поля
```

---

## D2 — дополнительные данные

Нужны иногда.

Примеры:

```text
activity feed
comments
history
settings details
attachments
```

Правила:

```text
D2 грузится отдельно/lazy
D2 не должен тормозить D0/D1
```

---

## D3 — тяжёлые/редкие данные

Примеры:

```text
audit log
export
old archive
analytics
full history
large reports
```

Правила:

```text
D3 отдельно
D3 часто async
D3 можно партиционировать
D3 можно уносить в отдельное хранилище
```

# 5. Базовая структура таблицы

Для бизнес-сущностей по умолчанию:

```sql
CREATE TABLE projects (
  id              UUID PRIMARY KEY,
  organization_id UUID NOT NULL REFERENCES organizations(id),
  owner_id        UUID NOT NULL REFERENCES users(id),

  name            TEXT NOT NULL,
  description     TEXT,
  status          TEXT NOT NULL CHECK (status IN ('active', 'archived', 'deleted')),

  version         INTEGER NOT NULL DEFAULT 1,

  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at     TIMESTAMPTZ,
  deleted_at      TIMESTAMPTZ
);
```

Базовые поля:

| Поле | Зачем |
|---|---|
| `id` | стабильный primary key |
| `organization_id / tenant_id` | владелец данных |
| `created_at` | когда создано |
| `updated_at` | когда изменено |
| `deleted_at` | soft delete, если реально нужен |
| `version` | optimistic locking |
| `status` | lifecycle-состояние |

# 7. ID strategy

## Варианты

| Тип | Когда использовать |
|---|---|
| `BIGSERIAL/BIGINT` | максимум скорости, внутренняя система |
| `UUID` | распределённые системы, публичные id |
| `UUIDv7/ULID` | лучше для сортировки по времени |
| `TEXT external_id` | если id приходит из внешней системы |

Рекомендация по умолчанию:

```text
UUID или UUIDv7 для публичных бизнес-сущностей.
BIGINT можно для внутренних high-load таблиц.
```

Важно:

> Не показывай внутренний автоинкремент там, где это может раскрывать количество объектов или давать предсказуемый перебор.

## Деньги

Плохо:

```sql

```

Хорошо:

```sql

```

```sql

```

Всегда:


Правило:


## Статусы

Для часто меняющихся статусов:

```sql
status TEXT NOT NULL CHECK (status IN ('open', 'in_progress', 'done'))
```


Правило:

# 9. Constraints — база должна защищать данные

Backend validation нужен, но DB constraints обязательны.

## Обязательно использовать:

```text
PRIMARY KEY
FOREIGN KEY
NOT NULL
UNIQUE
CHECK
DEFAULT
```

Пример:

```sql
CREATE TABLE organization_members (
  id              UUID PRIMARY KEY,
  organization_id UUID NOT NULL REFERENCES organizations(id),
  user_id         UUID NOT NULL REFERENCES users(id),
  role            TEXT NOT NULL CHECK (role IN ('owner', 'admin', 'member')),

  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),

  CONSTRAINT uniq_org_member_user UNIQUE (organization_id, user_id)
);
```

Зачем:

```text
Backend может ошибиться.
DB не должна позволить сломать факты.
```

# 11. Индексы

Индексы — это не украшение.  
Индекс должен иметь причину.

## 11.1. Правило индекса

У каждого индекса должен быть owner:

```text
Какой endpoint/query использует этот индекс?
```

Пример:

```text
Endpoint:
GET /projects?organizationId=...&status=active&sort=-updatedAt

Index:
idx_projects_org_status_updated
```

```sql
CREATE INDEX idx_projects_org_status_updated
ON projects (organization_id, status, updated_at DESC, id DESC)
WHERE deleted_at IS NULL;
```

---

## 11.2. Порядок колонок в composite index

Обычно:

```text
1. tenant / organization_id
2. equality filters
3. range filters
4. sort columns
5. id for stable cursor
```

Пример:

```sql
CREATE INDEX idx_tasks_project_status_updated
ON tasks (project_id, status, updated_at DESC, id DESC)
WHERE deleted_at IS NULL;
```

Для запроса:

```sql
SELECT id, title, status, updated_at
FROM tasks
WHERE project_id = $1
  AND status = $2
  AND deleted_at IS NULL
ORDER BY updated_at DESC, id DESC
LIMIT 25;
```

---

## 11.3. Не индексировать всё подряд

Каждый индекс стоит ресурсов:

```text
больше размер DB
медленнее INSERT/UPDATE/DELETE
больше RAM/cache pressure
дольше migrations
```

Правило:

> Индекс добавляется только под конкретный access pattern.

---

## 11.4. Foreign keys часто требуют индекс

Если часто делаешь:

```sql
WHERE project_id = ?
```

нужен индекс:

```sql
CREATE INDEX idx_tasks_project_id ON tasks(project_id);
```

Но если уже есть:

```sql
CREATE INDEX idx_tasks_project_status_updated
ON tasks(project_id, status, updated_at DESC);
```

отдельный `idx_tasks_project_id` может быть лишним.

# 12. Pagination

## По умолчанию — cursor/seek pagination

Плохо на больших таблицах:

```sql
LIMIT 25 OFFSET 100000;
```

Хорошо:

```sql
SELECT id, title, updated_at
FROM tasks
WHERE project_id = $1
  AND deleted_at IS NULL
  AND (updated_at, id) < ($2, $3)
ORDER BY updated_at DESC, id DESC
LIMIT 25;
```

Cursor содержит:

```json
{
  "updatedAt": "2026-06-08T12:00:00Z",
  "id": "task_123"
}
```

Правила:

```text
сортировка должна быть стабильной
id добавляется как tie-breaker
limit всегда ограничен
max limit обычно 100
```

# 13. Query contracts

Каждый важный запрос должен быть описан как контракт.

Пример:

```yaml
accessPattern: tasks.listByProject
owner: modules/tasks
endpoint: GET /api/v1/projects/{projectId}/tasks

priority: D1

params:
  projectId: uuid
  status: optional string
  cursor: optional string
  limit:
    default: 25
    max: 100

select:
  - id
  - title
  - status
  - assignee_count
  - updated_at

filters:
  - project_id = :projectId
  - status = :status
  - deleted_at IS NULL

sort:
  - updated_at DESC
  - id DESC

index:
  name: idx_tasks_project_status_updated
  columns:
    - project_id
    - status
    - updated_at DESC
    - id DESC

budget:
  p95Ms: 80
  maxRowsReturned: 100
  maxRowsScanned: 500
  maxPayloadKb: 50
```

Главное:

> Нет query contract — значит запрос будет расти хаотично.

# 14. SELECT rules

## Нельзя

```sql
SELECT *
FROM projects;
```

## Нужно

```sql
SELECT id, name, status, owner_id, updated_at
FROM projects
WHERE organization_id = $1
ORDER BY updated_at DESC
LIMIT 25;
```

Правила:

```text
1. Выбирай только нужные колонки.
2. List query не должен доставать detail-поля.
3. Большие TEXT/JSON поля не тащить в списки.
4. Файлы/BLOB не хранить прямо в основных таблицах.
5. Для тяжёлых данных — отдельная таблица или object storage.
```

# 15. DTO ↔ DB

Не отдавать DB row напрямую наружу.

DB row:

```text
projects.id
projects.organization_id
projects.owner_id
projects.internal_flags
projects.deleted_at
```

API DTO:

```json
{
  "id": "prj_123",
  "name": "Mobile App",
  "status": "active",
  "owner": {
    "id": "usr_1",
    "name": "Alex"
  }
}
```

Правило:

> DB schema — это внутренний контракт хранения.  
> API DTO — это внешний контракт продукта.  
> Они не обязаны совпадать 1:1.

## Write model

Для записи лучше нормализованная модель:


Плюсы:

```text
меньше дублей
проще consistency
лучше constraints
проще транзакции
```

Для частых тяжёлых чтений можно делать read model:

```text
project_stats

dashboard_counters
search_documents
```

Пример:

```sql
CREATE TABLE project_stats (
```

Правило:

> Денормализация разрешена, если есть owner, способ обновления и документация.

# 17. Counters and stats

Плохо:

```text
На каждый dashboard делать COUNT(*) по огромным таблицам.
```

Лучше:

```text
project_stats
organization_stats
daily_metrics
materialized views
background aggregation
```

```sql
SELECT open_tasks, done_tasks
FROM project_stats
WHERE project_id = $1;
```

Вместо:

```sql
SELECT COUNT(*)
FROM tasks

```

на каждом открытии страницы.

# 18. Transactions

## Правила транзакций

```text
1. Транзакции должны быть короткими.
2. Не делать HTTP-запросы внутри DB transaction.
3. Не выполнять долгие вычисления внутри transaction.
4. Блокировки брать как можно позже.
5. Коммитить как можно раньше.
6. Для побочных эффектов использовать outbox.
```

---

## Optimistic locking

Для частого редактирования:

```sql
UPDATE projects
SET name = $1,
    version = version + 1,
    updated_at = now()
WHERE id = $2
  AND version = $3;
```

Если обновлено 0 строк:

```text
PROJECT_VERSION_CONFLICT
```

# 19. Outbox pattern

Чтобы не терять события:

```text
DB transaction
├── insert/update main data
├── insert outbox_event
└── commit

worker
├── read unpublished outbox events
├── publish to queue
└── mark as published
```

Пример таблицы:

```sql
CREATE TABLE outbox_events (
  id             UUID PRIMARY KEY,
  event_type     TEXT NOT NULL,
  aggregate_type TEXT NOT NULL,
  aggregate_id   UUID NOT NULL,
  payload        JSONB NOT NULL,
  status         TEXT NOT NULL DEFAULT 'pending',
  attempts       INTEGER NOT NULL DEFAULT 0,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  published_at   TIMESTAMPTZ
);

CREATE INDEX idx_outbox_pending_created
ON outbox_events (status, created_at)
WHERE status = 'pending';
```

```sql
CREATE TABLE audit_events (
```

```sql
CREATE INDEX idx_audit_org_created
```

```text
audit append-only

```

# 21. Multi-tenant правила

Если есть организации/аккаунты, почти каждая бизнес-таблица должна иметь:

```text
organization_id / tenant_id
```

Пример:

```sql
CREATE INDEX idx_tasks_org_project_status
ON tasks (organization_id, project_id, status, updated_at DESC);
```

Правила:

```text
1. Не доверять frontend.
2. tenant filter обязателен.
3. permissions проверяются на backend.
4. Индексы часто начинаются с organization_id.
5. Нельзя случайно сделать cross-tenant query.
```

# 22. Soft delete

Soft delete использовать только если есть причина:

```text
restore
audit/legal
user mistake recovery
sync with external systems
```

Поле:

```sql
deleted_at TIMESTAMPTZ
```

Запросы:

```sql
WHERE deleted_at IS NULL
```

Индексы:

```sql
CREATE INDEX idx_projects_active_org_updated
ON projects (organization_id, updated_at DESC)
WHERE deleted_at IS NULL;
```

Правило:

> Если везде есть soft delete, но нет restore/lifecycle — это просто мусор в таблицах.

# 23. Data lifecycle

## Hot / Warm / Cold

| Тип | Что это | Где держать |
|---|---|---|
| Hot | часто читается/пишется | основная DB |
| Warm | редко меняется, иногда читается | основная DB / read replica |
| Cold | архив/история/export | archive tables/object storage |
| Analytics | отчёты, агрегации | OLAP/ClickHouse/BigQuery/etc |

---

## Retention policy

Для каждой тяжёлой таблицы нужно знать:

```text
сколько хранить?
кто владелец?
можно ли удалить?
нужно ли архивировать?
есть ли legal hold?
```

Пример:

```yaml
table: activity_events
retention:
  hot: 90 days
  archive: 2 years
  deleteAfter: 3 years
```

# 24. Partitioning

Партиционирование нужно не всегда.

Использовать, когда:

```text
таблица очень большая
данные естественно режутся по времени/tenant
часто удаляются старые периоды
запросы почти всегда содержат partition key
```

Хорошие кандидаты:

```text
audit_events
activity_events
logs
metrics
outbox history
large transactions
```

Плохие кандидаты:

```text
маленькие таблицы
таблицы без понятного partition key
таблицы, где запросы ходят по всем партициям
```

# 25. Migrations

Миграции должны быть безопасными.

## 25.1. Структура

```text
db/
├── migrations/
│   ├── 202606080001_create_projects.sql
│   ├── 202606080002_add_project_status.sql
│   └── 202606080003_create_tasks.sql
├── schema/
├── seeds/
├── fixtures/
├── access-patterns/
├── docs/
└── README.md
```

---

## 25.2. Expand / Contract pattern

Для zero-downtime изменений:

```text
1. Add new nullable column
2. Deploy app writes both old and new
3. Backfill data in batches
4. Deploy app reads new column
5. Add NOT NULL / constraints
6. Remove old column later
```

---

## 25.3. Чего избегать

Опасно:

```text
добавить NOT NULL column без default/backfill
переименовать колонку без совместимости
удалить колонку сразу
создать индекс blocking способом на большой таблице
огромный UPDATE без батчей
```

Для больших таблиц:

```text
backfill batches
concurrent indexes
lock timeout
statement timeout
migration dry-run
```

# 26. Resource budgets для DB

Стартовые бюджеты:

| Метрика | Default |
|---|---:|
| Simple primary key lookup | 1–10ms |
| Обычный indexed list query | до 30–80ms |
| Сложный read query | до 100–200ms |
| Slow query threshold | 100–200ms |
| Max rows returned list | 100 |
| Default list limit | 25–50 |
| Max rows scanned/returned ratio | желательно < 10:1 |
| Transaction duration | как можно меньше, обычно < 100–300ms |
| Migration lock timeout | 1–5s |
| HTTP request DB queries | 5–10 max |

Правило:

> Если endpoint быстрый, но DB читает 500k строк ради 25 результатов — это не быстрый endpoint, это будущая авария.

# 28. Read replicas

Read replica полезна для:

```text
тяжёлых read-only запросов
reports
analytics-ish screens
admin lists
exports
```

Но нельзя забывать:

```text
реплика может отставать
read-after-write может быть неправильным
права/критичные проверки лучше читать с primary
```

Правило:

> Если пользователь только что создал объект и сразу его открывает — читай с primary или учитывай consistency.

---

# 29. Files / blobs

Плохо:

```text
хранить большие файлы в основной OLTP DB
```

Хорошо:

```text
object storage: S3/R2/GCS/etc
DB хранит metadata
```

Таблица:

```sql
CREATE TABLE files (
  id              UUID PRIMARY KEY,
  organization_id UUID NOT NULL,
  owner_user_id   UUID NOT NULL,
  storage_key     TEXT NOT NULL,
  file_name       TEXT NOT NULL,
  mime_type       TEXT NOT NULL,
  size_bytes      BIGINT NOT NULL,
  checksum        TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

---

# 30. Search

Не пытайся превратить основную DB в поисковый движок для всего.

Можно в PostgreSQL:

```text
простые фильтры
простая полнотекстовая индексация
ILIKE на маленьких таблицах
```

Для сложного поиска:

```text
OpenSearch/Elasticsearch/Meilisearch/etc
```

Правило:

> OLTP DB — источник истины.  
> Search index — производная read model.

---

# 31. Analytics

Не нагружай основную DB тяжёлыми аналитическими запросами.

Если нужны:

```text
воронки
длинные отчёты
агрегации по миллионам событий
исторические графики
```

лучше:

```text
events table → pipeline → OLAP storage
```

Примеры OLAP:

```text
ClickHouse
BigQuery
Snowflake
Redshift
DuckDB for small internal analytics
```

Backend request:

```text
GET /api/v1/views/project-detail/{projectId}?sections=main,tasks,stats
```

Request Tree:

```text
project-detail
├── main
│   ├── project summary
│   ├── permissions
│   └── owner summary
├── tasks
│   ├── task list
│   └── assignees batch

    └── project_stats
```

```text
q1 projects.getSummaryById
q2 organization_members.getByUserAndOrg
q3 tasks.listByProject
q4 users.batchSummaryByIds
q5 project_stats.getByProject
```

Index Tree:

```text
projects:
  pk_projects_id

organization_members:

tasks:
  idx_tasks_project_status_updated

users:
  pk_users_id

project_stats:
  pk_project_stats_project_id
```

Вот так backend и DB становятся одной понятной системой.

# 34. Security baseline для DB

Обязательно:

```text
least privilege
отдельные DB users/roles
нет root/admin user в приложении
шифрование sensitive данных
backup encryption
audit опасных действий
no secrets in logs
PII classification
```

---

## Роли

Примерно:

```text
app_rw       — приложение read/write только нужные схемы
app_readonly — read-only для reports/admin
migration   — миграции
analytics   — только read к нужным таблицам/views
```

Правило:

> Приложение не должно подключаться к DB как superuser.

---

## Sensitive data

К sensitive относятся:

```text
password hash
tokens
email
phone
address
payment data
documents
private notes
IP/user agent
```

Правила:

```text
1. Пароли — только hash, никогда plain text.
2. Tokens — хранить hash или encrypted, если возможно.
3. Payment card data лучше не хранить самому.
4. PII не писать в обычные logs.
5. Доступ к sensitive columns должен быть ограничен.
```

# 35. Backups / Restore

Backup без restore-теста — это надежда, а не система.

Обязательно определить:

```text
RPO — сколько данных можно потерять
RTO — за сколько времени нужно восстановиться
backup frequency
backup retention
restore procedure
encryption
who has access
```

Пример:

```yaml
backup:
  rpo: 15 minutes
  rto: 2 hours
  retention:
    daily: 14 days
    weekly: 8 weeks
    monthly: 12 months
  restoreTest: monthly
```

```text
[ ] описаны access patterns
[ ] добавлены индексы под access patterns
[ ] нет ненужного JSON

[ ] есть rollback/forward strategy
[ ] есть тесты
[ ] обновлена документация
```

# 38. Definition of Done для query

Query готов, если:

```text
[ ] имеет имя
[ ] имеет owner
[ ] описан access pattern
[ ] выбирает только нужные колонки
[ ] имеет limit для list
[ ] использует cursor для больших списков

[ ] проверен EXPLAIN/ANALYZE

[ ] не делает N+1

```

# 39. Definition of Done для migration

Migration готова, если:

```text
[ ] имеет понятное имя
[ ] маленькая и атомарная
[ ] безопасна для production
[ ] не держит долгие locks
[ ] большие backfill идут батчами
[ ] есть план rollback или forward-fix
[ ] протестирована на копии данных
[ ] совместима с текущей и новой версией app
[ ] обновлены schema/docs
```

# 40. 50 senior-законов DB

## Модель данных

1. База хранит факты, а не случайные JSON-куски.  
2. Таблица должна иметь владельца.  
3. Связи должны быть явными.  
4. Foreign key — это не бюрократия, а защита данных.  
5. Не храни массив foreign keys в JSON.  
6. Деньги не хранятся в float.  
7. Время храни в `timestamptz`.  
8. Один статус — один смысл.  
9. Soft delete должен иметь причину.  
10. Audit log append-only.

## Запросы

11. `SELECT *` — почти всегда ошибка.  
12. Каждый list query имеет limit.  
13. Offset pagination на больших данных — будущая проблема.  
14. Cursor pagination требует стабильной сортировки.  
15. COUNT на большой таблице должен быть осознанным.  
16. List DTO не должен тащить detail-поля.  
17. Большие TEXT/JSON не должны попадать в списки.  
18. Query должен иметь имя.  
19. Query без budget — неизвестная стоимость.  
20. N+1 — архитектурный запах.

## Индексы

21. Индекс создаётся под access pattern.  
22. Индекс без владельца станет мусором.  
23. Слишком много индексов замедляет записи.  
24. Composite index начинается с самых селективных/equality-фильтров и tenant.  
25. Sort должен учитываться в индексе.  
26. Partial index полезен для active/deleted/status данных.  
27. Foreign key часто требует индекс.  
28. Индекс не лечит плохой запрос полностью.  
29. Индексы нужно мониторить.  
30. Неиспользуемые индексы нужно удалять осторожно.

## Транзакции

31. Транзакции должны быть короткими.  
32. Не делай внешние HTTP calls внутри transaction.  
33. Lock нужно брать поздно и держать мало.  
34. Optimistic locking лучше долгих блокировок для UI.  
35. Побочные эффекты после commit — через outbox.  
36. Idempotency key нужен для повторяемых write requests.  
37. Deadlocks нужно логировать и разбирать.  
38. Retry write operation должен быть безопасным.  
39. Не смешивай долгую аналитику и OLTP transaction.  
40. Consistency важнее красивого shortcut.

## Миграции и эксплуатация

41. Миграции — часть кода.  
42. Большая миграция без dry-run — риск.  
43. Backfill делается батчами.  
44. Удаление колонок — последний шаг, не первый.  
45. Backup без restore-теста — не backup.  
46. Production DB не место для ручных экспериментов.  
47. Read replica не гарантирует свежесть данных.  
48. Analytics лучше выносить из OLTP.  
49. Observability DB обязательна.  
50. Schema docs экономят месяцы жизни команды.

# 41. Главное правило DB-системы

Если сомневаешься, выбирай так:

```text
1. Целостность данных
2. Безопасность данных
3. Понятность модели
4. Быстрые hot queries
5. Минимум лишних строк/колонок
6. Безопасные migrations
7. Observability
8. Масштабируемость
```

# 42. Короткая финальная формула

Твой DB-подход:

> **Access-pattern-first DB design with domain data tree, strict constraints, query budgets, read models and migration discipline.**

По-русски:

> **База данных от дерева доступа: доменная модель + запросы + индексы + жизненный цикл данных.**

Самая короткая мысль:

```text
DB должна быть картой быстрых и безопасных путей к данным,
а не складом таблиц, которые когда-то кому-то понадобились.
```
